# AGENTS.md — phone_db_rust (phone_db)

Guidance for agentic coding agents operating in this repository.

## Project Overview

A Dioxus fullstack web application (Rust) for managing phone contacts and call routing. The frontend compiles to WASM; the backend runs as an Axum server. PostgreSQL is the primary store; LDAP mirrors contact data. App name in `Dioxus.toml`: `phone_db`. Package name in `Cargo.toml`: `phone_db`.

## Build & Run Commands

```bash
# Development server (web, hot-reload)
dx serve --platform web

# Production build (prefer via Nix)
dx build --release --verbose
nix build

# Server binary only (no WASM frontend)
cargo run --features server

# Tailwind CSS (run in parallel with dx serve)
npx tailwindcss -i ./input.css -o ./assets/tailwind.css --watch

# Diesel database migrations
diesel migration run

# Regenerate Diesel schema (after migration changes)
diesel print-schema > src/server/database/schema.rs
```

### Nix / devenv

```bash

```

## Test Commands

There are **no `cargo test` unit tests** in this project. All testing is done via a NixOS VM integration test in `flake.nix`.

```bash
# Run the full integration test (requires Nix)
nix flake check --impure

# The test spins up the full service stack and checks /_health via curl.
# There is no mechanism to run a single isolated test.
```

When adding new functionality, prefer verifying correctness by building the project and using the integration test rather than adding unit tests (unless the domain logic is complex and self-contained).

The codebase has two main Cargo features:

| Feature         | Purpose                                                       |
| --------------- | ------------------------------------------------------------- |
| `web` (default) | Client/WASM build                                             |
| `server`        | Enables all server-side deps (Axum, Diesel, LDAP, OIDC, etc.) |

## Architecture & Module Layout

```
src/
  main.rs            # Route enum (#[derive(Routable)]) + App root component
  lib.rs             # Re-exports models and server
  dt.rs              # Date/time utilities
  version.rs         # BUILD_DATE / VCS_REF from build env
  models/            # Shared data types (no DB deps; used client + server)
  functions/         # Dioxus #[server] functions (client sees only signature)
  views/             # Page-level Dioxus components (one per route)
  components/        # Reusable UI components
  forms/             # Form infrastructure: fields, validation, saving state
  server/            # Server-only code (#[cfg(feature = "server")])
    auth/            # axum-login Backend, session middleware
    database/
      schema.rs      # Auto-generated by diesel print-schema (do not edit)
      models/        # Diesel Queryable/Insertable structs + CRUD functions
      service/       # Business logic: coordinates DB + LDAP
    ldap/            # LDAP pool, filters, queries
    oidc/            # OpenID Connect client + middleware
```

### Three-Layer Database Pattern

All DB operations follow this strict layering:

1. **`server/database/models/xxx.rs`** — Raw Diesel structs and bare async CRUD. Returns `diesel::result::Error`. Schema DSL imports are scoped to each function body.
2. **`server/database/service/xxx.rs`** — Business logic in transactions. Combines DB + LDAP. Converts to shared `models::` types via `From`/`Into`.
3. **`functions/xxx.rs`** — Dioxus `#[server]` functions. Calls the service layer, converts errors to `ServerFnError`.

## Code Style Guidelines

### Naming Conventions

- Standard Rust: `snake_case` for functions/variables/modules/fields, `PascalCase` for types/traits/enums/structs.
- Dioxus component functions use `PascalCase` (requires `#![allow(non_snake_case)]` at top of component files).
- Domain ID newtypes: `XxxId(i64)` — e.g., `ContactId`, `UserId`, `PhoneCallId`.
- Insert structs: `NewXxx` (e.g., `NewContact`).
- Update structs (shared models layer): `ChangeXxx` (e.g., `ChangeContact`).
- Update structs (Diesel layer): `UpdateXxx` (e.g., `UpdateUser`).
- **Noun-first component naming**: `[Noun][Verb][Subverb]` — e.g., `ContactUpdateDialog`, `UserDeleteButton`.

### Naming Vocabulary (from README)

Use these verbs consistently:

| Verb                | Meaning                                |
| ------------------- | -------------------------------------- |
| `Add` / `Remove`    | Modify a list                          |
| `Create` / `Delete` | Create or permanently remove an entity |
| `Archive`           | Soft-hide an entity                    |
| `Update`            | Modify an existing entity              |
| `View`              | Display/read-only component            |
| `Change`            | Pending (unsaved) change state         |
| `New`               | Pending (unsaved) new entity state     |

### Imports

- Group order: `std` → external crates → `crate::` local.
- Server-only imports: wrap with `#[cfg(feature = "server")]` on individual `use` statements or modules.
- Schema DSL imports are deferred to the function body (not file-level) in Diesel query functions:
  ```rust
  use crate::server::database::schema::contacts::dsl as q;
  ```

### Error Handling

- Use `thiserror` (`#[derive(Error)]`) for all custom error types.
- Layer errors: each module has its own `Error` enum with `#[from]` for lower-level conversions.
- Error propagation chain: `diesel::result::Error` → `database::connection::Error` → `service::Error` → `AppError` → `ServerFnError`.
- Bridge to Dioxus clients: `.map_err(AppError::from).map_err(ServerFnError::from)`.
- Use `?` throughout; avoid manual `match` on `Result` when `?` suffices.
- Use `.pipe(Ok)` (from the `tap` crate) instead of wrapping the last expression in `Ok(...)`:
  ```rust
  table.select(...).filter(...).get_result(conn).await.optional()?.pipe(Ok)
  ```

### Types & Patterns

- Use **newtype IDs** for all domain identifiers; implement `FromStr`, `Display`, `Copy`, `Serialize`, `Deserialize`.
- Use **`MaybeSet<T>`** (defined in `models/common.rs`) instead of `Option<Option<T>>` for partial updates: `Set(T)` | `NoChange`.
- Use `chrono::DateTime<Utc>` for all database timestamps; `DateTime<FixedOffset>` or `DateTime<Local>` for user display/input.
- Reactive state: `Signal<T>` (mutable), `Memo<T>` (derived), `Resource<T>` (async).
- Component callbacks: `Callback<T>` props (Dioxus equivalent of event handlers).
- The `FieldValue` trait bridges raw string inputs ↔ typed values for forms.

### Transactions

Use pinned async closures for Diesel transactions:

```rust
conn.transaction::<_, Error, _>(|conn| Box::pin(async move {
    // ...
})).await
```

### Diesel Queries

Always scope schema DSL to the function body. Use type-safe builder DSL:

```rust
use crate::server::database::schema::contacts::dsl as q;
table
    .select(Contact::as_select())
    .filter(q::id.eq(id))
    .get_result(conn)
    .await
    .optional()
```

### LDAP Mirroring

Contact writes must be **mirrored to LDAP within the same transaction**. If LDAP fails, the transaction rolls back. Only write to LDAP when: contact has a name, a non-anonymous phone number, and `action == "allow"`.

### Dioxus Components

- Routes are a `#[derive(Routable)]` enum in `main.rs`.
- Dialog visibility is encoded in URL query params (e.g., `?dialog=update-42`) via `ListDialogReference` enums with `FromStr`/`ToString`.
- Current user: loaded once with `use_server_future`, stored in Dioxus context as `Resource<Result<Option<Arc<User>>, ServerFnError>>`.
- Async data fetching: `use_resource(|| async { ... })`.
- Form loading state: `Saving` enum — `No` / `Yes` / `Finished(Result<(), EditError>)`.

### Clippy & Formatting Annotations

- `#[rustfmt::skip]` — used on the `Route` enum; apply sparingly.
- `#![allow(non_snake_case)]` — required at the top of component files (Dioxus PascalCase components).
- `#[allow(dead_code)]` — acceptable on Diesel model fields not yet read in application code.
- `#[allow(clippy::to_string_trait_impl)]` — used where `ToString` is implemented directly.
